In industries if your pipeline processes "Bad" data the system fails, 
    - so engineers build "Gatekeepers" ensure only perfect data reaches to the model.

in production AI bad data is normal
    Examples:

    - broken sensors
    - missing temperature values
    - impossible pressure readings
    - corrupted timestamps
    - strings instead of numbers

    If invalid data reaches your model:

        - predictions become unreliable
        - pipelines crash
        - dashboards become misleading

The modern industries standard uses:
        - Pydantic
            used heavily with :
                    - Fast API
                    - AI API
                    - ML backend
                    - IOT

Traditional approach is :
    if age>0 and age< 120: 
        problem with it : - messy - repetitive - hard to scale

Professional approach:

class SensorData(BaseModel):
    temperature:float
    humidity:int 

        - Pydantic 
            Automatically 
                - validates types
                - check constraints
                - throw meaningful errors



machine_id:str
    temprature:float=Field(..., gt=-40, lt=125) #temperature range for industrial machines
    pressure:float=Field(..., gt=0) #pressure must be positive
    humidity:float=Field(..., ge=0, le=100) #humidity must be between 0 and 100

            ---------In Pydantic field constraints:

                            gt = greater than
                            lt = less than
                            ge = greater than or equal to
                            le = less than or equal to

what is 
            1] BaseModel:
                        - Every schema inherits from: BaseModel
                          This gives:
                                - validation
                                - parsing
                                - serialization
                                - error handling

            2] Field()

                    - Used for constraints.

----------------------Configuration Management (YAML/JSON)--------------------------------
Goal: Stop hardcoding paths and settings.

Professional German engineering focuses on reusability. You should never have to change your Python code just because a file moved to a different folder. Instead, you use Configuration Files.


-------PyYAML library to read these settings----------------------
This is a major step toward production-grade engineering.


Right now your code has:

pipeline = DataPipeline(
    input_dir="data/raw",
    output_dir="data/processed"
)


problem:
    - paths are hard encoded
    - changing path requires code edits
    - bad for collaboration
    - hard for deployment

professional system moves settings into:
    - YAML
    - JSON
    - environment variables

            -----this is called comfiguration Management-----

why this is matters:

 your pipeline can run on :
        - windows
        - linux
        - docker container
        - zeiss server
        - cloud vm 

    without changing code , only Configuration changes
    

all  related to this code is in day 3 file 


conofig.yaml file

input_dir: "path"
output_dir: "path"
